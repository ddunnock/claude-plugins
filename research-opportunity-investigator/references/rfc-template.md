# RFC Template for ACP Protocol

This template MUST be used for all RFCs generated by the Research & Opportunity Investigator skill.

## RFC Header

```markdown
# RFC-XXXX: [Title]

| Field | Value |
|-------|-------|
| RFC Number | RFC-XXXX |
| Title | [Descriptive Title] |
| Author | [Author Name/Handle] |
| Status | Draft |
| Created | [YYYY-MM-DD] |
| Updated | [YYYY-MM-DD] |
| Requires | [List any prerequisite RFCs, or None] |
| Supersedes | [List any RFCs this replaces, or None] |
```

---

## Required Sections

### 1. Summary

**Requirements:**
- MUST be 2-3 sentences maximum
- MUST clearly state what the RFC proposes
- MUST NOT include implementation details

**Template:**
```markdown
## Summary

This RFC proposes [specific change] to enable [specific capability]. 
The change affects [list of components] and provides [key benefit].
```

---

### 2. Motivation

**Requirements:**
- MUST include Problem Statement
- MUST include Research Basis (for research-originated RFCs)
- MUST include Gap Analysis (for integration RFCs)

**Template:**
```markdown
## Motivation

### Problem Statement

[What specific problem does this RFC solve? Be concrete.]

### Research Basis

This RFC is based on research conducted into [TARGET], specifically:

| Source ID | Source | Relevance |
|-----------|--------|-----------|
| [SRC-001] | [URL/Document] | [How it informed this RFC] |
| [SRC-002] | [URL/Document] | [How it informed this RFC] |

### Gap Analysis

This RFC addresses the following identified gaps:

| Gap ID | Description | ACP Benefit |
|--------|-------------|-------------|
| GAP-001 | [Description] | [How ACP addresses this] |
| GAP-002 | [Description] | [How ACP addresses this] |
```

---

### 3. Specification

**Requirements:**
- MUST list all affected components
- MUST show existing vs proposed state
- MUST provide rationale for each change
- MUST reference existing spec sections

**Template:**
```markdown
## Specification

### Overview

[High-level description of the proposed changes - 1-2 paragraphs]

### Affected Components

| Component | Type | Change Type | Description |
|-----------|------|-------------|-------------|
| [spec/chapters/XX.md] | Spec Chapter | [Add/Modify/Remove] | [What changes] |
| [schemas/v1/XX.json] | Schema | [Add/Modify/Remove] | [What changes] |
| [RFC-XXX] | RFC | [Extends/Supersedes] | [Relationship] |

### Detailed Changes

#### Change Area 1: [Name]

**Existing State** (from [spec chapter/RFC]):

\`\`\`
[Current specification text, schema fragment, or code]
\`\`\`

**Proposed State**:

\`\`\`
[Proposed specification text, schema fragment, or code]
\`\`\`

**Rationale**: [Why this change is needed - trace to gap analysis]

#### Change Area 2: [Name]

[Continue pattern for each change area]
```

---

### 4. Backward Compatibility

**Requirements:**
- MUST explicitly state breaking changes (or "None")
- MUST provide migration path if breaking
- MUST specify deprecation schedule if applicable

**Template:**
```markdown
## Backward Compatibility

### Breaking Changes

[List all breaking changes, or state "None - this RFC is fully backward compatible."]

### Migration Path

[If breaking changes exist, describe how existing users migrate.]
[If no breaking changes, state "Not applicable - no migration required."]

### Deprecation Schedule

[If anything is deprecated, provide timeline.]
[Otherwise state "None - no deprecations introduced."]
```

---

### 5. Implementation

**Requirements:**
- SHOULD describe reference implementation approach
- MUST include test cases

**Template:**
```markdown
## Implementation

### Reference Implementation

[Description of how this would be implemented in acp-cli or other reference tools]

### Test Cases

| Test ID | Description | Expected Result |
|---------|-------------|-----------------|
| TC-001 | [Test scenario] | [Expected outcome] |
| TC-002 | [Test scenario] | [Expected outcome] |
| TC-003 | [Test scenario] | [Expected outcome] |
```

---

### 6. Alternatives Considered

**Requirements:**
- MUST list at least 2 alternatives
- MUST explain why each was rejected

**Template:**
```markdown
## Alternatives Considered

| Alternative | Pros | Cons | Reason Rejected |
|-------------|------|------|-----------------|
| [Alt 1] | [List pros] | [List cons] | [Why not chosen] |
| [Alt 2] | [List pros] | [List cons] | [Why not chosen] |
```

---

### 7. Security Considerations

**Requirements:**
- MUST address security implications
- MAY state "None identified" if truly none

**Template:**
```markdown
## Security Considerations

[Describe any security implications of this RFC, or state "None identified."]
```

---

### 8. References

**Requirements:**
- MUST list all sources used
- MUST include access dates

**Template:**
```markdown
## References

| ID | Source | Type | Accessed |
|----|--------|------|----------|
| [SRC-001] | [URL] | [documentation/source_code/announcement] | [YYYY-MM-DD] |
| [SRC-002] | [URL] | [documentation/source_code/announcement] | [YYYY-MM-DD] |
```

```markdown
[Any supplementary technical information]
```

## RFC Numbering Convention

- RFCs are numbered sequentially: RFC-001, RFC-002, etc.
- Use leading zeros for single digits: RFC-001 not RFC-1
- Check existing RFCs before assigning number

## Status Values

| Status | Meaning |
|--------|---------|
| Draft | Initial proposal, under discussion |
| Review | Under formal review |
| Accepted | Approved for implementation |
| Implemented | Changes merged into spec |
| Rejected | Not accepted |
| Withdrawn | Author withdrew proposal |
| Superseded | Replaced by newer RFC |

## Validation Checklist

Before submitting RFC:

```
□ Summary is 2-3 sentences
□ Problem statement is concrete
□ All sources cited with [SRC-XXX]
□ All gaps traced to research
□ Affected components listed with spec references
□ Existing vs proposed state shown for each change
□ Breaking changes explicitly addressed
□ At least 2 alternatives considered
□ Test cases provided
□ Security considerations addressed
□ All references include access dates
```
