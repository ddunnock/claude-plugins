  <action>
Create `scripts/interface_agent.py` following the decomposition_agent.py pattern exactly (data prep in Python, no Claude calls, structured output through SlotAPI).

1. `discover_interface_candidates(api: SlotAPI) -> list[dict]`:
   - Query approved components: `api.query("component", {"status": "approved"})`
   - Query accepted component-proposals to get relationships: `api.query("component-proposal", {"status": "accepted"})`
   - For each accepted proposal with relationships, find the corresponding committed component via proposal.decision.committed_slot_id
   - For each relationship, find the target component by matching target_proposal name to committed component names
   - Deduplicate by sorted component pair (frozenset of source_id, target_id) -- user decision: one interface per pair
   - Also discover via requirement cross-references: two components sharing requirement_ids (from their parent_requirements or requirement_ids field) indicate a boundary. Filter out cross-cutting requirements that appear in 3+ components.
   - Return list of candidate dicts: {source_component, target_component, relationship_type, description, discovery_method ("relationship"|"requirement_crossref"), shared_requirement_ids}

2. `detect_orphan_components(api: SlotAPI, candidates: list[dict]) -> list[str]`:
   - Find approved components that appear in ZERO interface candidates
   - Return list of component slot_ids (Claude's discretion: report these for user awareness)

3. `class InterfaceAgent`:
   - `__init__(self, api: SlotAPI, schemas_dir: str)` -- same pattern as DecompositionAgent
   - `prepare(self) -> dict` -- calls discover_interface_candidates and detect_orphan_components, returns structured data dict for Claude to enrich with direction, protocol, data_format_schema, error_categories
   - `create_proposals(self, interfaces: list[dict], session_id: str) -> list[dict]` -- validates each interface dict, creates interface-proposal slots via SlotAPI. Each proposal gets: name (Claude-provided, convention "{source}-to-{target}: {purpose}"), description, source_component, target_component, direction, protocol, data_format_schema, error_categories, rationale, requirement_ids, proposal_session_id, status="proposed", gap_markers=[]
   - `format_preparation_summary(self, data: dict) -> str` -- human-readable summary of discovered candidates and orphans

4. `check_stale_interfaces(api: SlotAPI) -> list[dict]`:
   - Query approved interfaces: `api.query("interface", {"status": "approved"})`
   - For each interface, check if source_component or target_component has updated_at newer than interface's updated_at (user decision: timestamp-based detection)
   - Return list of stale interface dicts with slot_id, name, reason (which component changed)
   - During normal command runs, stale interfaces trigger automatic re-proposal (no separate --refresh flag per user decision)

5. Create `tests/test_interface_agent.py` with:
   - test_discover_candidates_from_relationships: 2 components with accepted proposals having relationships -> 1 candidate
   - test_discover_candidates_deduplicates_pairs: Bidirectional relationships produce 1 candidate, not 2
   - test_discover_candidates_from_requirement_crossrefs: 2 components sharing requirement_ids -> 1 candidate
   - test_crossref_filters_crosscutting_requirements: Requirement in 3+ components excluded from interface discovery
   - test_detect_orphan_components: Component with no relationships or shared requirements detected
   - test_create_proposals_validates_and_persists: InterfaceAgent.create_proposals creates valid interface-proposal slots
   - test_check_stale_interfaces_detects_changed_component: Interface with older updated_at than its component flagged as stale
   - test_check_stale_interfaces_none_stale: No stale interfaces when timestamps are current
   - Performance: test_discover_candidates_100_components runs under 1 second
