Add to `scripts/view_assembler.py`:

1. `assemble_view(api: SlotAPI, spec: dict, workspace_root: str, schemas_dir: str) -> dict` — The main assembly function:
   a. Call `capture_snapshot(api)` once to get consistent state (VIEW-07)
   b. Load gap rules via `load_gap_rules(workspace_root)`
   c. For each scope_pattern in spec["scope_patterns"]:
      - Call `match_scope_pattern()` against the snapshot
      - If matches found: apply field selection (if `fields` directive present), add to sections
      - If no matches: call `build_gap_indicator()` and add to gaps list
   d. Organize slots hierarchically: components as tree roots, interfaces nested under their parent component (using `source_component`/`target_component` fields), contracts under interfaces. Slots without a parent go in an "unlinked" section.
   e. Build metadata: spec_name, assembled_at (ISO timestamp), snapshot_id (from snapshot), total_slots, total_gaps, gap_summary (counts by severity)
   f. Validate output against view.json schema (VIEW-06) — use SchemaValidator
   g. Return the assembled view dict

   CRITICAL: This function ONLY reads from SlotAPI. It does NOT create, update, or delete any slots (VIEW-10). The assembled view is returned as a dict, not persisted (REQ-354, VIEW-01).

2. `_apply_field_selection(slots: list[dict], fields: list[str]) -> list[dict]` — Filter each slot dict to only include the specified fields plus always-included system fields (slot_id, slot_type, name, version).

3. `_organize_hierarchically(sections: list[dict]) -> list[dict]` — Reorganize flat section lists into tree structure. Components are roots. Interfaces with source_component/target_component matching a component slot_id are nested under that component. Contracts with interface_id matching an interface are nested under that interface. Everything else goes in "unlinked".

Add to `scripts/view_assembler.py`:

1. `BUILTIN_SPECS` — Dict mapping spec names to their spec dicts for the 5 built-in view types:
   - `system-overview`: scope_patterns for all components + their interfaces (slot_types: component, interface)
   - `traceability-chain`: requirements -> components -> interfaces -> contracts (all 4 types)
   - `component-detail`: parameterized by `{component_id}`, matches single component + all related interfaces and contracts
   - `interface-map`: all interfaces grouped by component pairs
   - `gap-report`: all slot types with scope pattern `*` to surface all gaps across the design

2. `render_tree(view: dict) -> str` — Render assembled view as human-readable indented tree:
   - Header: spec name, assembled at, total slots, total gaps
   - For each section: slot_type heading, indented slot entries showing name + key fields
   - Hierarchical: nested components > interfaces > contracts with 2-space indent levels
   - Gap indicators: prefixed with "[GAP]" marker, severity in brackets, suggestion on next line
   - Gap summary at bottom: counts by severity
   - Use Unicode box-drawing characters for tree connectors (e.g., `+-- `, `|   `)

3. `create_ad_hoc_spec(patterns: list[str]) -> dict` — Create a transient spec from ad-hoc inline scope patterns provided on the command line. Parses each pattern string as `<slot_type>:<name_glob>`, constructs a valid spec dict internally.

4. `get_builtin_spec(name: str, parameters: dict | None = None) -> dict` — Look up a built-in spec by name, resolve parameters if needed. Raise ValueError if name not found.

```
# /system-dev:view

Assemble a contextual view from registry slot subsets.

## Invocation

/system-dev:view <spec_name_or_pattern> [--json] [--param key=value]

Parameters:
- spec_name_or_pattern (required): Name of built-in spec (system-overview, traceability-chain, component-detail, interface-map, gap-report) OR an ad-hoc scope pattern (e.g., "component:Auth*")
- --json (optional): Output structured JSON instead of human-readable tree
- --param (optional, repeatable): Parameter for parameterized specs (e.g., --param component_id="Auth Service")

## Workflow
1. Determine spec: If spec_name matches a built-in, use it. Otherwise treat as ad-hoc pattern.
2. Resolve parameters: Apply --param values to parameterized specs.
3. Initialize: Load SlotAPI from .system-dev/ workspace.
4. Assemble: Call assemble_view() with the spec.
5. Render: If --json, output JSON. Otherwise call render_tree().
6. Display: Print the rendered output.

## Built-in Views
[List the 5 built-in specs with descriptions]

## Output Format
[Show example tree output and JSON output]

## Error Handling
- No workspace: Suggest /system-dev:init
- Unknown spec name + not a valid pattern: List available built-in specs
- Missing required parameter: Show which parameters are needed
- Empty view (no slots, no gaps): Show "View is empty" message
```
