  <action>
    Create scripts/traceability_agent.py with class TraceabilityAgent following the established agent pattern (data prep, no AI calls):

    1. **Constructor**: Takes SlotAPI instance.

    2. **TRACEABLE_TYPES**: List of all slot types to include as graph nodes: need, requirement, component, interface, contract (plus synthetic vv: nodes).

    3. **build_graph(api)** -> dict:
       - Collect edges from traceability-link slots via api.query("traceability-link")
       - Collect edges from embedded fields on committed slots:
         - component.requirement_ids -> edges from req_id to comp slot_id, type="allocated_to"
         - component.relationships -> edges between component pairs, type="communicates_with"
         - interface.source_component/target_component -> edges from comp to interface, type="boundary_of"
         - interface.requirement_ids -> edges from req_id to interface slot_id, type="allocated_to"
         - contract.component_id -> edge from component to contract, type="constrained_by"
         - contract.interface_id -> edge from interface to contract, type="constrained_by"
         - contract.obligations[].source_requirement_ids -> edges from req to contract, type="derives_from"
         - contract.vv_assignments -> create synthetic node "vv:{contract_id}:{assignment.obligation_id}" with edge from contract, type="verified_by" (Pitfall 6)
       - Deduplicate: edges with same (from_id, to_id, edge_type) keep one, prefer traceability-link source
       - Detect divergences: same (from_id, to_id) but different edge_type from different sources -> add to divergences list
       - Build forward adjacency dict and reverse adjacency dict (Pitfall 3)
       - Collect all nodes with type and name from slot data
       - Return graph dict matching traceability-graph schema structure

    4. **validate_chains(graph)** -> dict:
       - Expected chain levels: need(0) -> requirement(1) -> component(2) -> interface(3) -> contract(4) -> vv(5)
       - For each need node, walk forward through the chain levels using the adjacency dict
       - At each level, check if edges exist to the next level type. If not, record a gap at that level.
       - Three-level severity per CONTEXT.md:
         - Critical: chain fully broken (no path from need to any downstream element)
         - Warning: partial chain (path exists but stops before V&V)
         - Info: orphan element (not connected to any chain from either direction)
       - Orphan detection: after chain walking, find any node not visited from any need chain AND not reachable backward from any chain endpoint
       - Return {chains: [...], gaps: [...], divergences: [...]} plus completeness stats

    5. **check_staleness(api, graph_slot)** -> bool:
       - Compare graph_slot["built_at"] against max updated_at across all traceable slot types
       - Use the efficient approach from RESEARCH.md: check index-level metadata first if available, otherwise scan slot updated_at fields
       - Return True if any source slot is newer than built_at

    6. **build_or_refresh(api)** -> dict:
       - Try to read existing graph via api.read("tgraph-current") (well-known singleton ID)
       - If no graph exists, build new one, persist via api.create() with overridden slot_id="tgraph-current"
         - NOTE: Since SlotAPI.create() auto-generates IDs, use api.ingest() instead with deterministic ID "tgraph-current" and slot_type "traceability-graph"
       - If graph exists but stale, rebuild and persist via api.update()
       - If graph exists and fresh, return it
       - Always run validate_chains on the graph and include chain_report + completeness in the persisted slot

    7. **format_trace_output(graph_slot)** -> str:
       - Chain-per-need summary grouped by stakeholder need (per CONTEXT.md)
       - Completeness percentage at top of output
       - Breaks highlighted inline within each chain
       - Divergences in separate section (per CONTEXT.md -- not mixed with gaps)
       - Orphans listed at bottom as info severity

Use a temporary workspace with SlotAPI, ingest/create test slots representing a small design graph (2 needs, 3 requirements, 2 components, 1 interface, 1 contract with V&V), then exercise graph building and chain validation.
