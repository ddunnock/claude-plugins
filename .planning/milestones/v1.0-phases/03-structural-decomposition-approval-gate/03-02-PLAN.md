1. Create `scripts/decomposition_agent.py` with:

   - `detect_requirement_gaps(requirements: list[dict]) -> dict` -- analyze requirements for completeness BEFORE decomposition (Pitfall 6). Returns:
     - total_requirements: int
     - missing_descriptions: list of slot_ids with no/empty description
     - missing_source_blocks: list of slot_ids with no source_block
     - incomplete_slots: list of slot_ids that already have gap_markers
     - gap_count: total gaps (missing_descriptions + incomplete_slots)
     - severity: "high" if gap_count > 20% of total, "medium" if > 0, "none" if 0
     - Per CONTEXT.md: "Warn first when gaps are significant"

   - `check_stale_components(api: SlotAPI) -> list[dict]` -- proactive flagging when gaps are filled (per CONTEXT.md locked decision: "Proactive flagging when gaps are filled -- when new requirements are ingested, affected components are flagged for potential revision"). Logic:
     - Query all accepted component slots (status == "approved")
     - For each component, get its parent_requirements list (requirement slot_ids)
     - Query those requirement slots and check for changes since the component was last updated:
       - requirement has new/changed gap_markers compared to when component was accepted
       - requirement status has changed (e.g., was "active", now "deprecated" or "modified")
       - requirement updated_at is newer than the component's updated_at
     - Returns list of stale component dicts: {component_slot_id, component_name, stale_reason: str, affected_requirement_ids: list[str], component_updated_at, newest_requirement_updated_at}
     - Log warnings for each stale component via logger.warning
     - This function is called at the START of decompose workflow (before prepare) so the user sees which existing components may need revision before creating new proposals

   - `prepare_requirement_data(api: SlotAPI) -> dict` -- query all requirement slots, extract fields needed for decomposition (slot_id, upstream_id, description, requirement_type, source_block, parent_need, gap_markers, derives_from). Also query need slots for context. Returns:
     - requirements: list of extracted requirement dicts
     - needs: list of extracted need dicts
     - gap_report: result of detect_requirement_gaps
     - by_source_block: dict grouping requirements by source_block (for chunking context)

   - `DecompositionAgent` class:
     - `__init__(self, api: SlotAPI, schemas_dir: str)` -- store api and schemas_dir references
     - `prepare(self) -> dict` -- calls prepare_requirement_data, returns the data bundle. This is the data preparation step that happens before Claude does the reasoning.
     - `create_proposals(self, components: list[dict], session_id: str, agent_id: str = "structural-decomposition") -> list[dict]` -- takes the AI-generated component groupings (list of dicts with name, description, requirement_ids, rationale, relationships, gap_markers) and creates component-proposal slots via SlotAPI.create. Each proposal gets:
       - status: "proposed"
       - proposal_session_id: session_id
       - requirement_ids: list of requirement slot_ids
       - rationale: {narrative: str, grouping_criteria: list, evidence: list of {req_id, relevance}}
       - relationships: list of {target_proposal (to be linked after all created), type, description}
       - gap_markers: inherited from requirements + any new gaps from decomposition
       - decision: {action: null, decided_by: null, decided_at: null, notes: null, rejection_rationale: null, modifications: null, committed_slot_id: null}
     - Returns list of created proposal dicts with slot_ids.
     - `format_coverage_summary(self, proposals: list[dict], total_requirements: int) -> str` -- format "47/52 requirements mapped, 5 unmapped (gap markers)" summary per CONTEXT.md.
     - `format_proposal_narrative(self, proposals: list[dict]) -> str` -- format proposals as narrative blocks: component name + 1-line purpose + requirement count header, then rationale narrative, then evidence list. Per CONTEXT.md: "Summary-first density" and "Narrative blocks format."

   - Use `from scripts.registry import SlotAPI` for all persistence (XCUT-04).
   - Use `import logging; logger = logging.getLogger(__name__)` (XCUT-02).
   - The agent does NOT call Claude or generate prompts -- it prepares data and formats output. The actual AI reasoning happens in the command workflow (decompose.md) where Claude reads the prepared data and produces groupings.

1. Create `agents/structural-decomposition.md` -- agent definition for Claude:
   - Role: Analyze ingested requirements and propose component groupings
   - Input: Prepared requirement data from DecompositionAgent.prepare()
   - Instructions:
     - Group requirements by functional coherence and data affinity
     - AI-adaptive granularity: determine component count from natural clustering, not a fixed target
     - Flat component list (single level, no sub-hierarchy)
     - For each component produce: name, description (1-2 sentences), requirement_ids (list of slot_ids), rationale (domain narrative headline + evidence), relationships (data flows/dependencies to other proposed components)
     - For rationale: "domain narrative as headline with requirement IDs as supporting evidence" per CONTEXT.md
     - Single proposal set (not alternatives) unless ambiguity clearly warrants it
     - Detect and report any requirements that don't fit naturally into any component (these become gap markers)
   - Output format: JSON array of component objects matching the create_proposals input format
   - Gap handling: If gap_report.severity is "high", explain what's missing before proposing

2. Create `commands/decompose.md` -- command workflow:
   - Trigger: `/system-dev:decompose` command
   - Steps:
     1. Initialize workspace (scripts/init_workspace.py)
     2. Call check_stale_components(api) to detect accepted components whose requirements have changed. If any stale components found, show them to user with stale reasons and ask whether to proceed (per CONTEXT.md: "Proactive flagging when gaps are filled")
     3. Call DecompositionAgent.prepare() to get requirement data + gap report
     4. If gap_report.severity == "high": show gap summary, ask user whether to proceed
     5. Present requirement data to Claude using agents/structural-decomposition.md instructions
     6. Claude analyzes and produces component groupings
     7. Call DecompositionAgent.create_proposals() with Claude's groupings
     8. Show coverage summary (format_coverage_summary)
     9. Show proposals in narrative format (format_proposal_narrative)
     10. Inform user to run `/system-dev:approve` to review proposals
   - Reference scripts: scripts/decomposition_agent.py, scripts/registry.py, scripts/init_workspace.py

3. Create `commands/approve.md` -- command workflow:
   - Trigger: `/system-dev:approve` command
   - Steps:
     1. Initialize workspace
     2. Create ApprovalGate instance with SlotAPI and approval-rules.json
     3. Call gate.get_pending() to find actionable proposals
     4. If no pending proposals: inform user, exit
     5. Present proposals in batch (CONTEXT.md: "Batch review -- see all proposals, then decide on each")
     6. For each proposal, user chooses: accept, reject (with rationale), or modify (split/merge/edit)
     7. For modifications: support split (one proposal -> two), merge (two -> one), rename, scope changes per CONTEXT.md
     8. Call gate.decide() or gate.batch_decide() with user decisions
     9. Show results: which proposals accepted (with new component IDs), rejected, modified
     10. For rejections: auto re-propose -- re-run decomposition agent with rejection rationale as additional context (CONTEXT.md: "conversational loop")
     11. If modified proposals remain, loop back to step 5
   - Reference scripts: scripts/approval_gate.py, scripts/decomposition_agent.py, scripts/registry.py

4. Create `tests/test_decomposition_integration.py` with:
   - End-to-end test: ingest requirements -> prepare data -> create proposals -> accept via gate -> verify component slots exist
   - Test: create proposals -> reject one -> verify proposal status is rejected
   - Test: create proposals -> modify one -> accept modified -> verify component created from modified version
   - Test: gap detection with incomplete requirements -> proposals include gap markers -> accept with gaps -> component has gap markers
   - Test: coverage summary matches actual mapping (no requirement counted twice, unmapped requirements identified)
   - Test: stale component detection after re-ingestion -- accept a proposal, then update a referenced requirement's gap_markers, call check_stale_components, verify the component is flagged
   - Use real SlotAPI with tmp_path workspace, populate with realistic requirement/need slot data (at least 10 requirements across 3 source blocks).
