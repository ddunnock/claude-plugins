1. Create `scripts/approval_gate.py` with:

   - `load_approval_rules(rules_path: str) -> dict` -- load and return JSON config
   - `validate_transition(rules: dict, current_state: str, action: str, decision_data: dict) -> tuple[bool, str]` -- check if transition is valid per config, return (is_valid, error_message). Check terminal states, valid actions, required fields.
   - `ApprovalGate` class:
     - `__init__(self, api: SlotAPI, rules_path: str, proposal_type: str)` -- takes SlotAPI instance, path to approval-rules.json, and the proposal slot type name (e.g., "component-proposal"). The gate is generic -- it uses proposal_type for queries, not hardcoded types.
     - `decide(self, proposal_id: str, action: str, decision_data: dict, agent_id: str = "approval-gate") -> dict` -- main entry point. Loads proposal by ID, validates transition, executes action:
       - For "accept": Create committed slot FIRST (component from proposal data), then update proposal status/decision (Pitfall 2 ordering). The committed slot type is derived from the proposal type: strip "-proposal" suffix (e.g., "component-proposal" -> "component"). Map proposal fields to component fields: name, description, status="approved", parent_requirements=requirement_ids, rationale=rationale["narrative"].
       - For "reject": Update proposal status to "rejected", set decision.rejection_rationale from decision_data.
       - For "modify": Apply decision_data["modifications"] dict to proposal fields (shallow merge), update status to "modified", clear previous decision.
       - Always set decision.action, decided_by="developer", decided_at=UTC ISO, notes from decision_data.get("notes").
       - Per APPR-03: All SlotAPI writes (create + update) complete BEFORE returning result.
     - `batch_decide(self, decisions: list[dict], agent_id: str = "approval-gate") -> list[dict]` -- process multiple decisions sequentially. Each decision is {proposal_id, action, decision_data}. If one fails, stop and return results so far plus the error. Log batch journal entry via SlotAPI.
     - `get_pending(self) -> list[dict]` -- query proposals with status "proposed" or "modified" (actionable states).

   - IMPORTANT: The gate must NOT import from decomposition_agent or reference component-specific fields in its generic logic. The field mapping for "accept" uses a simple convention: proposal fields map to the committed slot type's schema. The gate reads the proposal's fields and maps them generically.

   - Use `from scripts.registry import SlotAPI` for all persistence (XCUT-04).
   - Use `import logging; logger = logging.getLogger(__name__)` for structured logging (XCUT-02).
