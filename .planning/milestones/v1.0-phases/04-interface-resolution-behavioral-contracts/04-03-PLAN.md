  <action>
Create `scripts/contract_agent.py` following the decomposition_agent.py pattern exactly.

1. `load_vv_rules(rules_path: str) -> dict`:
   - Load and return the vv-rules.json config file
   - Same pattern as load_approval_rules in approval_gate.py

2. `assign_vv_methods(obligations: list[dict], vv_rules: dict) -> list[dict]`:
   - For each obligation, look up obligation_type in vv_rules["default_methods"]
   - If found, assign that method with rationale "Default for {type} obligations per vv-rules.json", is_override=False
   - If not found, default to "test" with rationale "No default rule for {type}; defaulting to test", is_override=False
   - Return list of {obligation_id, method, rationale, is_override} dicts
   - This produces DEFAULTS -- Claude can override any of these with a different method and rationale (is_override=True)

3. `prepare_obligation_data(api: SlotAPI) -> dict`:
   - Query approved components: `api.query("component", {"status": "approved"})`
   - Query approved interfaces: `api.query("interface", {"status": "approved"})`
   - Query ingested requirements that map to each component (via component's requirement_ids/parent_requirements)
   - For each component, group its requirements by type/category and pair with its interfaces
   - Return structured dict with components, their interfaces, their requirements, organized for Claude to derive INCOSE-style obligations
   - Include gap markers for components with incomplete requirement mappings

4. `class ContractAgent`:
   - `__init__(self, api: SlotAPI, schemas_dir: str, vv_rules_path: str)` -- loads vv_rules on init
   - `prepare(self) -> dict` -- calls prepare_obligation_data, returns structured data for Claude
   - `create_proposals(self, contracts: list[dict], session_id: str) -> list[dict]`:
     * Each contract dict from Claude has: component_id, interface_id, name, description, obligations (list of {id, statement, obligation_type, source_requirement_ids, error_category?}), rationale
     * For each contract, call assign_vv_methods to get default V&V assignments
     * Merge Claude's vv_override list (if present) over defaults -- overrides replace matching obligation_id entries with is_override=True
     * Create contract-proposal slot via SlotAPI with all fields including bundled vv_assignments
     * Obligations use INCOSE wording: "SHALL process X within Y", "SHALL emit Z on failure" (user decision)
   - `format_preparation_summary(self, data: dict) -> str` -- human-readable summary

5. `check_stale_contracts(api: SlotAPI) -> list[dict]`:
   - Query approved contracts: `api.query("contract", {"status": "approved"})`
   - For each contract, check if its interface_id slot has updated_at newer than contract's updated_at
   - One-level cascade only: interface -> contract (user decision: no cascading back)
   - Return list of stale contract dicts with slot_id, name, reason

6. Create `tests/test_contract_agent.py` with:
   - test_assign_vv_methods_uses_defaults: data_processing obligation gets "test" method
   - test_assign_vv_methods_unknown_type_defaults_to_test: Unknown obligation_type gets "test" fallback
   - test_assign_vv_methods_all_types_covered: Each of the 8 default types produces correct method
   - test_prepare_obligation_data_groups_by_component: 2 components with different requirements produce separate groupings
   - test_prepare_obligation_data_includes_interfaces: Approved interfaces appear in component data
   - test_create_proposals_bundles_vv_assignments: Created contract-proposal has both obligations and vv_assignments
   - test_create_proposals_applies_ai_overrides: Claude override replaces default V&V for specific obligation
   - test_check_stale_contracts_detects_changed_interface: Contract with older updated_at than interface flagged stale
   - test_check_stale_contracts_no_cascade_back: Contract change does NOT flag interface as stale (one-level only)
   - Performance: test_assign_vv_100_obligations runs under 0.5 seconds
